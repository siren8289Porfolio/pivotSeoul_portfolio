# §10 최종 정리 · §11 ERD 개념과 읽는 법

## §10 최종 정리

기능별 ERD는 전체 ERD를 단순히 쪼갠 것이 아니라, 개발자가 실제로 구현할 수 있도록 **기능 책임 단위로 관계를 다시 그린 설계도**다.

- 공통 시뮬레이션 기능은 세션·시나리오·실행·결과를 담당한다.
- 주거 기능은 RIR와 주거 지속 임계점을 담당한다.
- 커리어 기능은 학습 가능 시간과 교육/일자리 추천을 담당한다.
- 보육/복직 기능은 돌봄 접근성과 복직 순이익 임계점을 담당한다.
- 노년 기능은 자산수명과 복지시설 접근성을 담당한다.
- 정책/RAG 기능은 추천 근거와 실행 액션을 담당한다.
- 데이터 출처 기능은 결과 재현성과 출처 투명성을 담당한다.
- LLM/로그 기능은 계산과 설명의 역할 분리를 검증한다.

이 문서 세트는 상위 문서 `ERD v4 — PRD·파일구조·폴더구조 통합`의 하위 상세로 쓰면 된다.

## §11.1 ERD 개념

ERD(Entity Relationship Diagram)는 DB 테이블과 관계를 시각화한 설계도다.

| 개념 | 설명 | 기능별 ERD 예시 |
| --- | --- | --- |
| Entity | 저장 대상. 보통 DB 테이블이 된다. | `SCENARIO_RESULT`, `RECOMMENDATION_CANDIDATE` |
| Attribute | 테이블이 가지는 컬럼. | `fit_score`, `risk_score`, `threshold_status` |
| PK | 각 행을 구분하는 기본키. | `scenario_result_id` |
| FK | 다른 테이블을 참조하는 외래키. | `threshold_result.scenario_result_id` |
| 1:N 관계 | 하나가 여러 하위를 가진다. | 하나의 `SCENARIO_RESULT`는 여러 `THRESHOLD_RESULT`를 가진다. |
| 도메인 | 기능 책임 단위. | 주거, 커리어, 보육, 노년, 정책/RAG |

## §11.2 기능별 ERD가 필요한 이유

1. **개발 범위 분리**: 기능별로 봐야 할 테이블만 분리해 볼 수 있다.
2. **담당자 협업**: Backend / AI / Frontend가 같은 기능 단위로 대화한다.
3. **변경 영향 파악**: 계산식 변경 시 어떤 결과·로그 테이블이 영향받는지 알 수 있다.
4. **테스트 설계**: 기능별 테스트 데이터와 검증 기준을 만들기 쉽다.
